## What this probe says

1. **The shim is solid** — second confirmed run past permissions on
   API 28; full 45 s execution; 5166 production sink events observed.
2. **The rejected-carrier finding holds on T390** — the receiver
   instrumented test ingests no MB cue + no direct-MX envelope from the
   same emit that successfully transmitted on R52. Same pattern as
   iPhone→R52 in recapture-3.
3. **New observation needing a follow-up at some point:** even R52→T390
   over the air didn't produce a decoded MB cue on the receiver. Two
   plausible causes — bench RF / positioning, or Android-9 ScanRecord
   parsing differing from Android 13's in a way that hides the
   manufacturer-data shape MeshX uses. Not a blocker for the carrier
   decision (which is already locked as rejected); flag it for a
   future T390-specific receive-path triage if T390 ever becomes a
   first-class production target.

The carrier-decision conclusion is unchanged: **MB legacy beacon +
GATT fetch remains the only validated production iOS↔Android route**
(see `docs/ble_carrier_decision.md`). The T390-receive negative is a
secondary data point, not a new blocker.
